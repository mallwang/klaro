# Implementation Plan: Forgot Password

**Branch**: `021-forgot-password` | **Date**: 2026-06-14 | **Spec**: [spec.md](spec.md)

**Input**: Feature specification from `specs/021-forgot-password/spec.md`

## Summary

Add a "forgot password" flow to the sign-in page. Users can request a password reset link via email, click the link, and set a new password. The feature reuses the existing `email_verifications` table with a new `purpose` column to distinguish email-change tokens from password-reset tokens. The flow follows the established token-generation, email-sending, and confirmation patterns already in the codebase.

## Architecture Decisions

### Decision 1: Reuse `email_verifications` table with `purpose` column

**Options considered**:
1. Reuse `email_verifications` table, add `purpose` column (`'email-change' | 'password-reset'`)
2. Create new `password_reset_tokens` table

**Decision**: Option 1 ‚Äî extend `email_verifications` with a `purpose` column.

**Rationale**:
- Consistent with spec requirement FR-008 ("reuse existing email verification approach")
- Avoids table proliferation for a simple token-based flow
- Single `UNIQUE INDEX` on `(user_id, purpose)` enforces one active token per user per purpose
- Existing `requestEmailChange` and `confirmEmailChange` methods continue to work unchanged
- Migration is additive (ALTER TABLE ADD COLUMN) ‚Äî no data loss

**Trade-offs**:
- Slightly more complex queries (must filter by `purpose`)
- But this is trivial ‚Äî a single WHERE clause

### Decision 2: Token expiry = 1 hour (not 24 hours)

**Rationale**:
- Password reset links are higher-risk than email verification links
- 1 hour is standard for password reset (GitHub, Google, etc.)
- Email verification links can be 24 hours because they're less sensitive

### Decision 3: No separate rate limiting for forgot-password endpoint

**Rationale**:
- Existing brute-force lockout (5 attempts ‚Üí exponential lockout) already protects sign-in
- Forgot-password endpoint only sends emails, doesn't reveal account existence
- Personal-use app with small user base ‚Äî YAGNI for dedicated rate limiting

### Decision 4: Generic success messages for email enumeration protection

**Rationale**:
- FR-007 requires generic messages regardless of email existence
- Both "email exists" and "email not found" return same response: "If an account exists, a reset link has been sent."
- Prevents attackers from enumerating valid email addresses

## Data Model

### Schema change: `email_verifications` table

```sql
ALTER TABLE email_verifications ADD COLUMN purpose TEXT NOT NULL DEFAULT 'email-change'
  CHECK(purpose IN ('email-change', 'password-reset'));
```

Update unique index to include purpose:

```sql
DROP INDEX IF EXISTS idx_email_verifications_user;
CREATE UNIQUE INDEX idx_email_verifications_user
  ON email_verifications(user_id, purpose);
```

## API Contracts

### POST /api/auth/forgot-password (public)

**Request**:
```json
{
  "email": "user@example.com"
}
```

**Response (202 Accepted)**:
```json
{
  "message": "If an account exists with that email, a password reset link has been sent."
}
```

**Errors**:
- 400 Bad Request: `{ "statusCode": 400, "error": "Bad Request", "message": "Invalid email" }`

**Notes**:
- Always returns 202, regardless of whether email exists (email enumeration protection)
- If email exists: generates token, sends email, returns 202
- If email not found: returns 202 without sending email

---

### POST /api/auth/reset-password/:token (public)

**Request**:
```json
{
  "password": "new-secure-password"
}
```

**Response (200 OK)**:
```json
{
  "message": "Password has been reset successfully"
}
```

**Errors**:
- 400 Bad Request: `{ "statusCode": 400, "error": "Bad Request", "message": "Password must be at least 8 characters" }`
- 404 Not Found: `{ "statusCode": 404, "error": "Not Found", "message": "Invalid or expired reset link" }`
- 410 Gone: `{ "statusCode": 410, "error": "Gone", "message": "This link has expired" }`

**Notes**:
- Token is single-use ‚Äî deleted after successful password reset
- Invalid or expired tokens return 404 (not 410) to avoid leaking token validity
- Password is validated against strength requirements (min 8 chars)

## UI Flow

### Step 1: Sign-in page

1. User sees sign-in form with email, password, and submit button
2. Below the form, "Forgot password?" link appears
3. Clicking link navigates to `/forgot-password`

### Step 2: Forgot password page

1. User sees email entry form with title "Reset your password"
2. User enters email and submits
3. Generic success message: "If an account exists with that email, a password reset link has been sent."
4. Message appears regardless of email validity (email enumeration protection)

### Step 3: Email received

1. User receives email with reset link
2. Link format: `{APP_URL}/reset-password/{token}`
3. Email subject: "Reset your password"
4. Email content: "You requested a password reset. Click the link below to set a new password."

### Step 4: Reset password page

1. User clicks link, navigates to `/reset-password/{token}`
2. User sees password entry form with title "Set a new password"
3. Form fields: "New password" (min 8 chars), "Confirm password"
4. Client-side validation: passwords must match
5. On submit: POST `/api/auth/reset-password/{token}` with new password

### Step 5: Success

1. Success message: "Your password has been reset."
2. "Sign in" button appears
3. User navigates to sign-in page

## Security Considerations

### Email Enumeration Protection

- Both "email exists" and "email not found" return identical 202 response
- Server logs may differentiate, but client never sees the difference
- Timing attack mitigation: always perform token generation (even for non-existent users), but skip email send

### Token Security

- 32-byte random token (256 bits of entropy)
- Single-use: deleted after successful password reset
- 1-hour expiry: shorter than email verification (24h) due to higher risk
- Previous tokens invalidated when new one requested (via DELETE WHERE user_id)

### Password Strength

- Minimum 8 characters (matches existing `ChangePasswordBodySchema`)
- Maximum 200 characters (prevents abuse)
- Same requirements as sign-up and password change

### Abuse Prevention

- No rate limiting beyond existing brute-force lockout (YAGNI)
- Token is single-use, so replay attacks are impossible
- Expired tokens are cleaned up on access (lazy cleanup)

## Migration Plan

### Database changes

1. Add `purpose` column to `email_verifications` table with default `'email-change'`
2. Drop existing unique index on `user_id`
3. Create new unique index on `(user_id, purpose)`
4. Existing rows get `purpose = 'email-change'` automatically

### Rollback strategy

- Column addition is additive ‚Äî can be reverted by dropping the column
- Index change is non-destructive ‚Äî can be reverted by recreating original index
- No data loss risk

### Deployment order

1. Deploy backend with migration (additive schema change)
2. Deploy frontend with new pages and routes
3. No downtime required ‚Äî migration is backward-compatible
