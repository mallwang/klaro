# Implementation Plan: Email Enhancements

**Branch**: `018-email-enhancements` | **Date**: 2026-06-14 | **Spec**: [spec.md](spec.md)

**Input**: Feature specification from `specs/018-email-enhancements/spec.md`

## Summary

Extend the existing `MailerService` with three new email methods (test, welcome, email-change
confirmation), wire them into the invitation-acceptance and email-change-confirmation routes
(fire-and-forget), expose a new admin-only `POST /api/admin/email/test` endpoint, and add a
"Test email delivery" section to the admin UI in `AccountsAdmin.tsx`.

No new database tables are needed. The only data-model change is widening the
`ConfirmEmailChangeResult` type in `ProfileService` to carry the confirmed new email address.

## Technical Context

**Language/Version**: TypeScript 5 (strict mode), Node.js LTS (ESM)

**Primary Dependencies**: Fastify (backend), React + Vite (frontend), Mantine v7 (UI),
nodemailer (already in use), Zod (already in use for request validation), `@pcm/shared`
(monorepo shared package)

**Storage**: SQLite via `better-sqlite3` — no schema changes required

**Testing**: Vitest (unit + integration), Playwright (e2e)

**Target Platform**: Linux server / WSL2 local dev

**Project Type**: Web application — pnpm monorepo with `packages/backend`, `packages/frontend`,
`packages/shared`

**Performance Goals**: Standard web-app expectations; email delivery is async and does not
affect response latency for the primary action

**Constraints**: Fire-and-forget for post-action emails (welcome, confirmation) — SMTP errors
must never fail or roll back the primary operation

**Scale/Scope**: Personal-use, small user base; no queue or retry infrastructure needed

## Constitution Check

*GATE: Must pass before Phase 0 research. Re-check after Phase 1 design.*

### Principle I — Test-First (NON-NEGOTIABLE)

**Gate status: PASS**

- Three new `MailerService` methods → unit tests written first in
  `mailer.service.test.ts` (stub transport pattern already established)
- `ProfileService.confirmEmailChange` return-type change → update existing unit tests first
- New `POST /api/admin/email/test` endpoint → integration test in new `admin.route.test.ts`
  written first
- Route-level mailer call-through (welcome email, confirmation email) → mock-mailer
  integration tests written first in existing `invitations.route.test.ts` and
  `profile.route.test.ts`

### Principle II — Type Safety (NON-NEGOTIABLE)

**Gate status: PASS**

- `ConfirmEmailChangeResult` widened to a discriminated union — fully typed, no `any`
- New `SendTestEmailBodySchema` in `@pcm/shared` — Zod-inferred type shared between backend
  and frontend
- New `sendTestEmail` frontend service function — typed return `Promise<void>`, error typed
  as `AuthError`
- `MailerService` new methods — `Promise<void>` return, `MailerError` on failure

### Principle III — Simplicity / YAGNI (NON-NEGOTIABLE)

**Gate status: PASS**

- No new npm packages
- No new database tables
- No retry queue or background worker
- Shared schema added to existing file (not a new file)
- Admin test-email UI added to existing admin page (not a new route/page)
- New backend route file `admin.ts` justified: keeps email-testing concerns separate from
  user-management (`users.ts`) and per-user profile (`profile.ts`)
